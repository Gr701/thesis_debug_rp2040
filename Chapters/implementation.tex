\section{Virtual Machine and Container}
To consistently replicate the software environment needed for debugging across diverse computing infrastructures, two approaches were considered: Virtual Machines and Containers.

To ensure objective evaluations, the underlying operating system was standardised as Debian 12 (Bookworm) across both the containerised and virtualised implementations. The distribution was chosen for its lightweight footprint and stability, and for its proven use in the Computer System course with Xubuntu, a Debian-based operating system \cite{tkj_vm_history}.

A functional debugging setup requires a software stack running on top of the operating system. The software stack used can be divided into four categories:

\begin{itemize}
    \item \textbf{Compiler Toolchain:} The suite of softwares needed to compile C/C++ source code into machine-readable binaries (e.g., UF2, ELF). The compile toolchain's components used for this thesis are \texttt{gcc-arm-none-eabi} cross-compilation infrastructure, which includes \texttt{libnewlib-arm-none-eabi} and \texttt{libnewlib-dev}, used as they provide the standard C library implementation, including memory management, mathematical functions, and system call stubs required for embedded I/O operations. To support the compilation infrastructure, \texttt{CMake} serves as the build system, and \texttt{Ninja} as the generator. Together, they provide a versatile and reliable framework for the Computer System course project and for more advanced embedded systems projects.
    
    \item \textbf{Middleware and SDKs:} To unlock all the features supported by the Raspberry Pi Pico platform and complex task management features, the environments also came prepared with Pico and FreeRTOS SDK. These provide the essential hardware abstraction and real-time scheduling capabilities required to support the course's embedded systems projects.
    
    \item \textbf{Development Interface:} To provide a familiar but versatile interface for project development, VS Code was selected for its ecosystem and flexibility, serving as the standardised IDE for the course. Alongside the main IDE, helper extensions are installed to provide effective interaction between the IDE and the development platform:
    
    \begin{itemize}
    \item \textbf{Cortex-Debug:} Facilitates GDB-based debugging, enabling step-through execution and register-level inspection.
    \item \textbf{Raspberry Pi Pico:} Provides essential SDK management and project configuration utilities.
    \item \textbf{Pico-Debug-Dashboard:} A custom-made extension with the intention of unifying how VS Code invokes the compiler toolchain and speaks with the hardware. Supports features such as:
    
    \begin{itemize}
    \item \textbf{Automated Project Bootstrapping:} Instant initialization of standardized project structures, including \texttt{CMake}, \texttt{FreeRTOS}, and debug configurations.
    \item \textbf{Dynamic Build Configuration:} Real-time switching between RP2040 and RP2350 target architectures and debug modes (1-core vs 2-core SMP) via the status bar.
    \item \textbf{Integrated Build Orchestration:} Single-click access to \texttt{Ninja} build processes, \texttt{GDB} flash deployment, and \texttt{Picotool} operations.
    \item \textbf{Hardware Lifecycle Management:} Centralised controls for hardware resets, BOOTSEL re-entry, and low-level firmware injection (\texttt{pico-debug} RAM-loading).
    \item \textbf{System Telemetry:} Real-time visual monitoring of USB bus states and debugger connectivity directly from the IDE status bar.
    \end{itemize}

    \end{itemize}
 As graphical interfaces are not natively supported in Docker containers, Code-Server is used in place of the original VS Code to provide a browser-based equivalent, ensuring a consistent user experience while maintaining the isolation and reproducibility of containerised environments. One significant difference between the container and VM implementations is the IDE-integrated serial monitor feature. While the standard Visual Studio Code desktop environment can access the official Microsoft Marketplace, \texttt{code-server} is limited to the Open VSX Registry. Consequently, some extensions, such as the Microsoft Serial Monitor, are unavailable in the web-based environment. To maintain a consistent workflow, Minicom is used as a terminal-based alternative to provide consistent monitor support for both implementations, with one-click status bar initialisation supported by the Pico-Debug-Dashboard extension. Microsoft Serial Monitor is still available on VMs as a fallback if problems with Minicom are encountered. 
    
    \item \textbf{Connectivity and Drivers:} The software components that bridge the host machine with the microcontroller. This includes OpenOCD as the GDB-to-CMSIS-DAP server and USB/HID drivers (such as \texttt{libhidapi} and \texttt{libusb}) required to maintain communication with the debug probe.
    
 Because virtualisation layers lack native access to USB devices, access must be managed by the virtualisation Infrastructure itself. While VM software supports USB passthrough natively, for Docker containers, a piece of software called \texttt{usbipd} must be installed on the host device.\ \texttt{usbipd} is a terminal software to redirect the physical USB devices from a Windows host to a guest virtual environment or the Windows Subsystem for Linux (WSL2). It functions by encapsulating USB traffic within TCP/IP packets, thereby overcoming the physical device isolation inherent in hypervisor-based architectures and allowing targeted devices to be seamlessly mounted within the execution environment. To make \texttt{usbipd} easier to use, a specialised script is provided to help users initialise and manage the tool.
\end{itemize}

All software components, including toolchain binaries, SDK releases, and debugging middleware, were configured according to their respective official documentation and established best practices and efforts were made to harmonise the configuration parameters across both the virtual machine and containerised environments. This is to ensure that any observed differences in performance or usability could be attributed strictly to the underlying virtualisation architecture.

\section{Metrics description}

Since there are several candidates for the implementation of the probe, we will define a set of metrics to choose the best one. Our set will include both quantitative and qualitative metrics, chosen based on the goal of the project, which is to implement a solution used throughout the educational process. While the values of the quantitative metrics will come directly from the tests and measurements, for qualitative metrics each of the team members will decide on a score from 1 to 5 and the average of those will be used in further considerations.

\subsection{Quantitative metrics}

The first metric considered is the \textbf{price of the hardware}. That is, the price of the plastic or other materials used for the casing, which would fit the TKJHAT and the probe together, as well as the price of the additional microcontrollers, boards, and wires used in the design. Since tens of debugger units are required by the course, the price of each one is a significant detail. 

Next factor we will consider is the \textbf{time to produce} the probe, which includes the printing time of the case for the given probe and the time for soldering and other assembling activities.   

The third quantative metric is the \textbf{physical size} of the final result. That is the size of the casing together with the TKJHAT and the probe. We will also consider the virtual size as a separate metric. It can be refered as \textbf{debug overhead} metric and it estimates the load on the CPU, memory and time consumption added by the debugger. While it might be one the most crucial metrics when working on big and complicated projects, it only takes secondary position in our research, since most of the projects at the entry level course are quite simple and compact.



    Slowdown Factor: Often measured in production debugging, this is the ratio of execution time with the debugger active vs. inactive (e.g., a "10x–100x slowdown" in intensive debugging scenarios).

    Memory Access Speed: The speed at which a debugger reads/writes data to the target system, crucial for embedded processors.

    Breakpoint Latency: Time taken to halt execution when a breakpoint is hit. 

\subsection{Qualitative metrics}

Qualitative metrics play a more important role in our project. Among these is how \textbf{well-maintained} the software to be used is: how often it is updated, if at all; how many people are behind it and how reliable it is in the long run. It is important since the outcome of our work is planned to be in use for a long time, about seven years, and it is likely, that there will be new bugs found and changes made in the upstream software.

One of the most crucial criteria is how \textbf{beginner-friendly} the setup and the usage of the probe is, how likely are the people, working with the embedded systems for the first time, to make a mistake, how easy it is to explain the workflow.

The general \textbf{comfort of use} is another metric we will consider. The hardware setup will affect our ratings as well as the response speed. We consider speed here, in the qualitative category, as exact miliseconds are not so important to us as the general feel and the smootheness of the process.

As a separate and final metric we will consider the \textbf{limitations} of each approach, which might not show in the rest of the metrics.


\section{Probe implementations}
\subsection{Second RP2040/RP2350 implementation}

The second board implementation utilises an additional RP-based microcontroller device as a dedicated hardware debugger. Now referred to as the probe, the device is flashed with the \texttt{debugprobe.uf2} firmware, turning it into a composite USB device that supports both the ARM CMSIS-DAP protocol for debugging and a Communication Device Class (CDC) interface for serial console monitoring. Physical connections between the probe and the target device are established via a 5-pin debug header, providing the necessary SWD signals and the bidirectional UART bridge needed for console output (see Box \ref{box:picoprobe-wiring} on page \pageref{box:picoprobe-wiring}). Once the connection between the probe and the host machine is established via USB, the debugging session can be initialised through VS Code or by manually initialising the GDB-to-OpenOCD-to-Probe pipeline through a terminal environment with commands\cite{pico_getting_started}:

\begin{tcolorbox}[
    colback=black!5,     
    colframe=black!75,   
    title=OpenOCD Initialization Command, 
    arc=0mm,
    boxrule=0.5pt,
    left=2mm
]
\texttt{\$ sudo openocd -f interface/cmsis-dap.cfg -f target/rp2040.cfg -c "adapter speed 5000"}
\end{tcolorbox}

For a RP2350-based device use \texttt{-f target/rp2350.cfg} instead.

Subsequently, the \texttt{GDB} client is invoked to link the debug symbols to the remote target:

\begin{tcolorbox}[title=GDB Client Initialization, colback=gray!5, colframe=black!75]
\begin{verbatim}
$ gdb-multiarch  <project_name>.elf
(gdb) target remote localhost:3333
\end{verbatim}
\end{tcolorbox}
Once the debugging pipeline is properly initialised, the platform provides a robust environment for system diagnostics, supporting a comprehensive set of \texttt{GDB} and \texttt{OpenOCD} features for hardware-level control:
\begin{itemize}
\item \textbf{Instruction-Level Control:} Enables precise program flow management through \texttt{step-into}, \texttt{step-over}, and \texttt{continue} operations, allowing the developer to isolate logic errors at the assembly or source code level.
\item \textbf{Hardware and Software Breakpoints:} The platform automatically switches between hardware-assisted breakpoints for Flash memory and software breakpoints for RAM to facilitate non-intrusive execution halting.
\item \textbf{System State Inspection:} Grants deep visibility into the processor by exposing the Program Counter (\texttt{PC}), Stack Pointer (\texttt{SP}), and general-purpose registers.
\item \textbf{Real-Time Memory and Register Analysis:} Allows for the dynamic examination and modification of memory addresses and peripheral register maps (\texttt{SVD} integration).
\item \textbf{Stack Traceback and Backtracing:} Provides the ability to traverse the function call hierarchy, which is critical for identifying recursive errors, stack overflows, and improper task transitions within a multi-threaded \texttt{FreeRTOS} environment.
\end{itemize}

As a primary diagnostic interface, \texttt{Minicom} is integrated into the development platform to ensure reliable serial communication across all environments. The monitor session is initialized via a standardized task-based command, which automatically detects the active communication port (e.g., \texttt{/dev/ttyACM0} or \texttt{/dev/ttyACM1}) and defaults to a baud rate of 115200 bps. Within the \texttt{Minicom} terminal, standard keyboard shortcuts provide access to the control menu and session termination. This terminal-based approach provides a platform-agnostic method for capturing low-level system logs and \texttt{printf} output, functioning reliably even when graphical serial interfaces are inaccessible.

To start serial monitoring inside of VS Code when Minicom fails to start, the Microsoft's Serial Monitor extension is installed to provide intuitive UI/UX for serial communication, enabling developers to monitor \texttt{UART} or \texttt{USB} standard output directly within the IDE. The monitor session is initialised by navigating to the serial monitor tab resides on the bottom panel, then choosing the target communication port, which is usually \texttt{/dev/ttyACM0} or \texttt{/dev/ttyACM1}, and matching the baud rate to the microcontroller's configuration, typically 115200 bps. To send messages to the target, a message prompt is presented at the bottom for convenient input.

\subsection{Second core implementation \- pico-debug}

The approach where one core is used to debug another one was implemented and shared by majbthrd on Github. He developed two projects: "pico-debug" for RP2040 \cite{pico_debug}, and "pico2-debug" for RP2350 boards \cite{pico2_debug}. To setup the debugging session there are ready .uf2 files provided in the repositories. Pico-debug can be placed in RAM or cache memory of the board, and each option has its own .uf2 file. We are going with a more common and easier way to use it, placing it in the RAM, so the whole cahce is availiable for the code of the program that is being debugged. To do so, we take the pico-debug-gimmecache.uf2 file from the releases page of the repository and flush it onto the board. 

Now the board shows as a CMSIS-DAP probe in the list of devices, and we can proceed to connect to it through the GDB and OpenOCD. The OpenOCD, however, has to be built by the user, since the newer versions of its Raspberry Pi's fork are not compatible with pico-debug. The detailed instructions are provided in the repository's "howto" section \cite{pico_debug}.

\subsection{3D-printed Casing}
Designing and producing a 3D-printed casing solution was paramount to ensuring that the Single Board Computer (SBC) and HAT board hardware was intergrated into a compact and durable debugging unit. The casing neeeded to be ergonomic, cheap and intuitive to use while providing the electronic components inside protection from damage that might occur during the usage of the  debugging unit. This requirement is reflected in the reinforced ridges along the chassis of the final casing that reduces the risk of cracking along the length of the case that might occur to accidental drops or abrasions.

Parametric design using CAD software was therefore required to ensure that the design met the stringent requirements required to provide students with an effective debugging solution. The design philosophy used in this project prioritsed simplity and comfort above all else, and this is reflected in the final version of the case produced. For instance, a simple inverted dovetail lid was implemented in the design to replace a complex hinge-and-lock system that was present in earlier iterations. 

The overall cumulative cost of raw materials required for manufacturing represented another significant and non-trivial consideration that was carefully evaluated throughout the iterative design process, given that there existed a clearly identified need to potentially scale the deployment of the debugging platform. After careful evaluation of available manufacturing materials, and due to the favorable combination of affordability characteristics and relative structural durability properties, Polylactic Acid (PLA) thermoplastic filament was selected and designated as the chosen material from which the protective enclosure would be fabricated and produced. The design was additionally and deliberately optimized specifically for ease of manufacturing using commercially available and widely accessible fused deposition modeling (FDM) three-dimensional printing equipment that is commonly present in many professional fabrication facilities, educational institutions, and even home-based maker and fabrication set-ups. Through careful iterative optimization of the design geometry and thoughtful selection of print parameters, the final optimized design iteration achieved a remarkably efficient combined printing time of approximately two hours for the complete assembly and required only 26 grams of PLA filament material for the entire protective casing structure, metrics that clearly indicate and demonstrate a high level of successful optimization with regard to both print settings configuration and manufacturing best practices.
