\section{Raspberry Pi and Pico family}

In this thesis we will focus on the computing platforms developed by Raspberry Pi Holdings. The first predecessors of the modern boards, which we will look at, were developed by the Raspberry Pi Foundation with the educational purposes, but due to its low cost, compact size, and flexibility, the boards quickly gained recognition and popularity in other domains as well \cite{raspberry_history}. Nowadays, the Raspberry Pi products have a wide range of appliencies: from hobby projects to industrial automation, robotics and Internet of Things (IoT) \cite{raspberry_success}. 

The Pico family is the group of Raspberry Pi microcontroller boards. It started in 2021 with Pico 1 series, which is built around the RP2040 silicon platform and includes 264KB on-chip RAM, 2MB on-board flash and Dual-core Arm Cortex-M0+ running at 133MHz frequency \cite{raspberry_pico_1}. The first series includes Pico, Pico H, Pico W, and Pico WH boards, where H stands for the presoldered headers, and W for the present wireless connectivity option. In 2024 the family got an update with the second series, including Pico 2 and Pico 2 W boards. New microcontrollers have twice the amount of RAM and flash of the first series, and the ability for the users to choose between the dual Arm Cortex-M33 and the dual RISC-V Hazard3 architechtures both running at 150MHz \cite{raspberry_pico_2}. All Pico family boards are programmable in C and MicroPython.

\section{Debugging theory}

The debugging is the integral part of the software development process. Its purpose is to ensure proper functioning and the performance of the software. There are different aproaches to debugging, with the simplest one being just printing the program values to the output terminal. In this thesis however, we will focus on the automated debugging with the extended control over the program's execution, which allows the programmer to use the break points, step one line at a time through the code, and monitor the values of the registers, variables, and expressions throughout the process.

The debugging process applied to the embedded devices is different, than the one used for the programs running on the host's CPU. That is because the debugger needs to control and therefore reach the processing unit of the microcontroller. Below we will take a look at how the control commands are forwarded from the host environment to the CPU on the microcontroller. In other words, how the debugging pipeline is implemented for the embedded devices and, in particular, Pico family microcontrollers and and Arm processors. \subsection{GDB}

For general C or C++ projects, the process of automated debugging is typically centered around the GNU Debugger (GDB) \cite{gdb_manual}. The main purpose of GDB is to provide the developer with a comprehensive and deep insight into the inner working of a program when it's running or to provide clues of what happened that led to the program's crash \cite{gdb_manual}.

\begin{figure}[ht]
    \centering
    \includegraphics[width=0.4\textwidth]{gdb_structure.png} 
    \caption{GDB internal structure \cite{gdb_architecture}.}
    \label{fig:gdb_structure}
\end{figure}

GDB's internal structure is described as containing two sides: the Symbol Side and the Target Side, with the Interface level sitting on top to act as the bridge between the developer and the program \cite{gdb_architecture}. This Interface level includes a Command Line Interface (CLI) for direct user interaction and a Machine Interface (MI) designed to be used by Integrated Development Environments (IDEs), such as Visual Studio Code (see Figure \ref{fig:gdb_structure}) \cite{gdb_architecture}.

The Symbol Side of \texttt{GDB} is built upon the Binary File Descriptor (\texttt{BFD}) library \cite{gdb_architecture}. \texttt{BFD}'s role is to act as the abstraction layer for a wide array of executable formats including \texttt{a.out}, \texttt{COFF} (Common Object File Format), and \texttt{ELF} (Executable and Linkable Format), allowing \texttt{GDB} to access raw symbolic data blocks regardless of the host's Operating System (\texttt{OS}) or the target's specific binary format \cite{gdb_architecture}. After \texttt{BFD} has navigated the executable, the task is then handed to the two levels of reader functions within \texttt{GDB}\cite{gdb_architecture}. These functions bridge the basic symbols used for name-to-address mapping with detailed metadata from formats such as \texttt{DWARF} or \texttt{stabs}, providing the high-level context of variable types, source line numbers, and processor register mappings necessary for debugging \cite{gdb_architecture}.

The Target Side of \texttt{GDB} is responsible for handling the dynamic execution of the program and raw data \cite{gdb_architecture}. To do that, GDB utilises a special class called Target Vector, a structure composed of function pointers with the purposes ranging from reading and writing of memory and registers, to setting parameters for handling shared libraries \cite{gdb_architecture}. The usage of the vectors is managed by a Target Stack, which sorts the vectors into prioritised groups to ensure vectors of running processes are always used while having the vectors of executables as fallbacks \cite{gdb_architecture}.

To tailor to the needs of a universal and portable debugger, \texttt{gdbarch} objects were introduced to encapsulate the architectural specifications, ranging from registers maps and the sizes of the different kinds of data to the calling convention and trap exceptions \cite{gdb_architecture}. This approach helps GDB's core logic stay independent of the underlyinghardware architecture, which allows GDB to be used on a diverse range of processors \cite{gdb_architecture}.

Sitting at the heart of GDB is the execution control unit, which has evolved from a monolithic loop called \texttt{wait\_for\_inferior} into a modular Event-Driven State Machine \cite{gdb_architecture}.This change has allowed GDB to becapable of managing complex state changes, trap instructions, and asynchronous debugging \cite{gdb_architecture}. 

To facilitate debugging on a secondary device, GDB makes use of the Remote Serial Protocol (RSP), which is a lightweight, packet-based communication standard \cite{gdb_architecture}. By using RSP, GDB can connect and communicate with the target hardware through either on-chip stubs or external translation programs known as sprites \cite{gdb_architecture}.

The debugging with GDB begins by compiling the program with the "-g" flag, which will generate, besides the executable, a \texttt{DWARF} table for the debugger \cite{gdb_manual}. With the generated table, the developer can control the program's execution through defined breakpoints and stepping. Bugs can also be isolated by analysing the program state and backtraces \cite{gdb_manual}.

In the scope of this thesis, a more specialised setup is required to cross-debug from a host device to the Raspberry Pi Pico \cite{pico_getting_started}. Toolchains such as \texttt{arm-none-eabi-gdb} for the \texttt{RP2040} or \texttt{riscv64-unknown-elf-gdb} for the \texttt{RP2350} are needed tobecause they contain the specific \texttt{gdbarch} modules necessary to interpret the registers and instruction sets of the target microcontrollers \cite{pico_getting_started}. Combining with OpenOCD, which serves as a specialized GDB server, this setup translates \texttt{RSP} packets into physical debug signals, allowing the developer to control the remote hardware as a local process \cite{pico_getting_started}.

\subsection{OpenOCD}

Originating as a diploma thesis work of Hubert Hogl and Dominic Rath, OpenOCD was created to bridge the gap between the cross-compiled GDB and the JTAG adaptor \cite{openocd_website}\cite{hoegl2006openocd}. OpenOCD is implemented as a server program that provides \texttt{GDB RSP} network socket on default port 3333 and a Telnet network socket on default port 4444 for manual interactive hardware control \cite{hoegl2006openocd}. It is also capable of utilising configuration files with target-specific configuration commands to accessthe target processor over one of several JTAG hardware interfaces, whilemaintaining log data \cite{hoegl2006openocd}.

\begin{figure}[ht]
    \centering
    \includegraphics[width=0.4\textwidth]{openocd_env.png}
    \caption{OpenOCD environment\cite{hoegl2006openocd}.}
    \label{fig:openocd_env}
\end{figure}

As illustrated in Figure \ref{fig:openocd_env}, a critical part of OpenOCD is the "monitor" command connection between the internal GDB server and the Telnet server, which included an internal command interpreter. This internal bridge allows developers to use any command that is available to a human userat the Telnet prompt, such as manual chip resets or flash bank configurations, without requiring a separate network connection \cite{hoegl2006openocd}.

The internal logic of OpenOCD serves as a protocol translator \cite{hoegl2006openocd}. OpenOCD takes software commands from GDB/Telnet and translates theminto binary bitstreams understandable by the targeted silicon \cite{hoegl2006openocd}. Translation and calculation of checksums for data integrity is all handled by the internal \texttt{main()} execution routine that serves as the system's scheduler \cite{hoegl2006openocd}.

The most integral module of the \texttt{main()} execution routine is the Target Module, which is described as the most complex component of the system's architecture \cite{hoegl2006openocd}. This module incorporates machine-level logics of supported target processors, including their register sets, memory map, and reset behaviours \cite{hoegl2006openocd}. The module determines the steps necessary to carry out a request, such as halting,stepping or accessing peripheral registers \cite{hoegl2006openocd}.

Once the Target Module has determined the instructions, they are passed to the interface layer, which historically required raw bit-banging to manually toggle the physical pins of a \texttt{JTAG} probe, it has evolved to support the modern CMSIS-DAP protocol (see 2.2.3) \cite{cmsis_dap, openocd_website,hoegl2006openocd}. Ultimately, the interface layer is the final stage of the software translation pipeline, responsible for converting the Target Module's logical instructions into the specific hardware-level commands and protocol-compliant data required to communicate with the target's physical debug port \cite{hoegl2006openocd}.

\subsection{Probe}

The next component in the pipeline is the probe. It serves as the translator between the digital commands - USB packets from OpenOCD (or another GDB server), and the electrical signals, which are then sent over the wires connected to the pins of the debugging port of the microcontroller board. There are protocols to standardize this communication, and one of them is CMSIS-DAP - an open standard described by Arm \cite{cmsis_dap}. It stands for the Common Microcontroller Software Interface Standard - Debug Access Port. Additionally to the protocol, it provides the template implementation of the concept \cite{cmsis_dap}. Its illustraion can be seen in \figref{fig:cmsis_dap_interface}. DAPLink is another implementaion of CMSIS-DAP. Moreover, there are proprietary protocols and their implementations, such as ST-Link by STMicroelectronics and J-Link by Segger \cite{zephyr_probes}.

The hardware part of the probe can be implemented in two major ways. It can be either a separate board, or a second microcontroller integrated into the board, which is being debugged \cite{zephyr_probes}. To communicate with the host machine, the probe is connected to it via USB, IP network, or other means \cite{zephyr_probes}.

\subsection{ADI}

Arm Debug Interface (ADI) architecture provides an access to the debug components inside the SoC (System on a Chip) \cite{adi}. It consists of Access Ports (APs), connected directly to the resources like CPU debug registers, memory system, ROM table, and more; and Debug Port (DP), which serves as a facade of the system and has a physical connection to the debug probe \cite{adi}. DP provides a one standard way to access the information from different APs \cite{adi}. \figref{fig:adiv5_implementation} illustrates the ADI version 5 with the block diagram. There are several protocols defining the implementation of the debug port. We will go through two of them: SWD and JTAG, in the following sections.

\subsection{SWD}

Serial Wire Debug (SWD) is the protocol used for Serial Wire - Debug Port (SW-DP) implementation of the DP. It is a serial, packet-based interface utilizing two pins: one for the bidirectional data connection, and the other one is for the clock, to synchronize the communication \cite{adi}. The operation on the wire has three main phases: packet request from the probe, acknowledgement response from the ADI, and the optional data transfer, either from the probe to ADI or vice versa \cite{adi}. Since all the communication happens on one line, there are turnaround periods between the phases, when one party has released the data line already, but the other party didn't take the control over it yet. It prevents the conflicts and during these periods the state of the line is undefined \cite{adi}.

The frame sent over the data line from the probe during the request phase of SWD is 8 bits long and has the information about the address of the register being accessed and whether it is a read or a write operation \cite{adi}. The acknowledgement frame is only 3 bits long: one per OK, WAIT, and FAULT statuses, while the data frame is 32 bits long \cite{adi}.

\subsection{JTAG}

Joint Test Action Group (JTAG) is the protocol used for JTAG Debug Port (JTAG-DP) implementation of the DP. The protocol utilizes four mandatory pins: DBGTDI (Debug Test Data In) for input data, DBGTDO (Debug Test Data Out) for output data, TCK (Test Clock) for synchronization, and DBGTMS (Debug Test Mode Select) for controlling state transitions \cite{adi}. An optional fifth pin, DBGTRSTn, provides asynchronous reset functionality \cite{adi}.

When a DAP is implemented with JTAG-DP, the wire-level interface provides access through two scan chains: the APACC (Access Port Access) scan chain for accessing debug resources in the system being debugged, and the DPACC (Debug Port Access) scan chain for accessing information internal to the DAP itself \cite{adi}. All state transitions occur synchronously with the clock, eliminating the need for turnaround periods as in SWD, since separate pins are used for input and output data \cite{adi}.

The data is shifted through the device serially via the DBGTDI and DBGTDO pins, with the DBGTAPSM providing two main paths: one for accessing the instruction register (IR) and another for accessing data registers (DR) \cite{adi}. The IR scan chain selects and controls the available scan chains, while the DR scan chain carries the actual data being transferred \cite{adi}.

\section{Options and approaches - software}

The development of embedded systems requires a complete toolchain environment including compilers, debuggers, software development kits, and integrated development environments. Providing a consistent, reproducible development environment is essential to minimise setup difficulties and reduces time spent troubleshooting configuration issues.

Two primary approaches exist for delivering such environments: containerisation through Docker and full virtualization through Virtual Machines. Each approach offers distinct advantages and disadvantages in terms of resource usage, hardware access, portability, and ease of deployment. The following subsections examine both approaches in detail, discussing their working principles, required dependencies, and the specific challenges encountered.

\subsection{Docker Containerisation}

Docker provides an isolated, lightweight environment for running the debugging software by leveraging Linux kernel features such as \texttt{namespaces} and \texttt{cgroups} to create containers. The containerisation approach packages the complete debugging toolchain, including OpenOCD, GDB, the Pico SDK, and Code-Server IDE, into a single reproducible image that can be deployed consistently across different host systems.

Docker containers share the host operating system's kernel, making them significantly lighter than full virtual machines. The container runs as an isolated process with its own file system, network stack, and process tree, while still executing directly on the host's hardware. For embedded debugging, USB devices must be explicitly passed through to the container using the \texttt{--privileged} flag and volume mounting (\texttt{-v /dev/bus/usb:/dev/bus/usb}), which grants the container direct access to the host's USB bus.

\subsection{Virtual Machine}

A Virtual Machine (VM) provides a complete, isolated operating system environment running on top of a hypervisor. Unlike Docker containers that share the host kernel, a VM includes its own kernel and can run a different operating system than the host. For the debugging platform, a Debian-based VM with full desktop environment can be configured using open-source tools used to build and manage VM environments, like Vagrant and VirtualBox.

VirtualBox creates a hardware-virtualized environment where the guest OS (Debian) runs with its own virtual CPU, memory, storage, and devices. The hypervisor intercepts hardware access requests from the guest and either emulates them in software or passes them through to real hardware. USB devices can be attached to the VM through VirtualBox's USB controller which provides native hardware access.

\subsection{second core debugger}

RP2040 chip has two Arm cores and provides an ability to control the SWD interface pins of both Core 0 and Core 1 from inside the chip \cite{raspberry_rp2040}. It means that we can manipulate the debug process of one core with the software running on the second core. Therefore, this approach eliminates the need for the external hardware. Its implementation involves a dedicated register - DBGFORCE, in the SYSCFG block of registers, and it has 8 bits - 4 bits per core, to control the debugging \cite{raspberry_rp2040}. The bits include:

\begin{itemize}
    \setlength\itemsep{0pt}
    \setlength\parskip{0pt}
    \item PROC\_ATTACH - to connect the DP of the core to syscfg control instead of the external SWD,
    \item PROC\_SWCLCK - to directly drive the clock,
    \item PROC\_SWDI - to drive the input on the SWDIO pin, and
    \item PROC\_SWDO - to observe the output on the SWDIO pin \cite{raspberry_rp2040}.
\end{itemize}

\subsection{Black Magic Probe}

The Black Magic Probe (BMP) is part of an open-source debugging solution, Black Magic Debug (BMD), that implements a GDB server directly in the probe hardware, eliminating the need for intermediate software like OpenOCD. It connects to the host computer via a standard USB interface, appearing as a CDC-ACM device (virtual serial port), which requires no special drivers on Linux or macOS \cite{blackmagic}. Unlike traditional debugging setups where a separate GDB server runs on the host machine, the BMP embeds the GDB Remote Serial Protocol server in its firmware, allowing direct connection from GDB to the probe's serial port \cite{blackmagic}.

The probe supports both JTAG and SWD protocols for connecting to ARM Cortex-M and Cortex-A based microcontrollers. When connected to a target device, the BMP provides full debugging functionality including hardware-assisted breakpoints (up to 6), watchpoints (up to 4 read, write, or access), memory and register examination, and flash memory programming \cite{blackmagic}. The firmware runs on dedicated Black Magic Probe hardware or can be flashed onto compatible devices such as ST-Link adapters \cite{blackmagic}.

The debugging workflow with Black Magic Probe is simplified compared to traditional setups \cite{blackmagic}. After connecting the probe to the host computer, GDB connects directly to the probe's serial port using the \texttt{target extended-remote} command \cite{blackmagic}. The probe automatically handles target detection through commands like \texttt{monitor swdp\_scan}, which identifies connected microcontrollers and reports their type and voltage \cite{blackmagic}. Once attached to a target, all standard GDB commands can be used to control program execution, examine state, and modify memory \cite{blackmagic}. The probe also includes a secondary USB-to-serial adapter, enabling simultaneous debugging and serial communication with the target device \cite{blackmagic}.

\subsection{JTAG/SWD J-Link Segger Probe}

J-Link is a family of debug probes developed by SEGGER Microcontroller GmbH that support both JTAG and SWD protocols for debugging ARM-based microcontrollers and other architectures \cite{jlink}. J-Link probes contain dedicated hardware with a built-in 32-bit RISC CPU that handles the protocol conversion and communication with the target device which discards the need for a separate GDB server running on the host computer \cite{jlink}.

J-Link probes support advanced debugging features including Real Time Transfer (RTT) for bidirectional communication with the running application, adaptive clocking that automatically adjusts to the target's clock speed, and Serial Wire Output (SWO) for instruction trace and event logging \cite{jlink}. Higher-end models support unlimited flash breakpoints, allowing developers to set as many breakpoints as needed in flash memory without being limited by the number of hardware breakpoint units available in the target processor \cite{jlink}. The probes also support multi-core debugging, where multiple debugger instances can access different cores in a JTAG scan chain simultaneously using a single J-Link probe \cite{jlink}.

\section{Computer Systems course and the TKJHAT}

521287A Introduction to Computer Systems is a degree level course that explains how a computer works as a complete stack, moving from high-level programs down to hardware in a conceptual and beginner-friendly way. It is a foundational course that is essential for prospective students wishing to advance to 521286A Computer Systems where the concepts are delved at a much deeper and more technical level, focusing on how these mechanisms are implemented and how they affect performance and functionality. While separate courses, they share the same course content at first as the two are implemented simultaneously, with students of both courses using the TKJHAT to experiment and derive solutions to complex problems presented to them during the implementation time of the courses.

The TKJHAT itself is an open-source, open-hardware, Hardware Attached on Top (HAT) platform designed by members of the Faculty of Information Technology and Electrical Engineering to support IoT and embedded systems education in the University of Oulu through the aforementioned courses. The development of this platform was driven by specific requirements identified through systematic evaluation: affordability (under 60€ per student), customizability to meet evolving teaching needs, easy integration into multiple courses, alignment with industry standards, Real-Time Operating System (RTOS) support, and wireless connectivity capabilities \cite{kelhala:2024}. 

The platform selection process involved a comprehensive systematic review of scientific literature and discourse analysis, evaluating alternatives including the nRF52-DK, STM Nucleo-64 family, Raspberry Pi Pico W, and Micro:bit v2 \cite{bscstudies}. The Raspberry Pi Pico W was ultimately selected as the base platform due to its optimal balance of price, functionality, strong community support, extensive input/output connections, wireless connectivity options, and compatibility with open-source development tools \cite{bscstudies,kelhala:2024}. 

The HAT platform was developed on top
of the Raspberry Pi Pico W and FreeRTOS with a diverse and multifaceted array of peripheral devices built in to extend the Pi from a general-purpose computer into a real-world measurement and control system that students can use effectively. FreeRTOS was selected as the RTOS after evaluating several alternatives including Zephyr RTOS, MbedOS, Kiel, and Riot \cite{kelhala:2024}. The selection criteria prioritized: (1) intuitive functions that expose hardware aspects without excessive abstraction, (2) thorough and easily accessible documentation, (3) ease of IDE installation and configuration with straightforward debugging capabilities, and (4) support for basic OS concepts such as tasks, mutexes, and I/O operations \cite{kelhala:2024}.

FreeRTOS emerged as the preferred choice due to its simplest API, excellent documentation with detailed usage examples, widespread industry adoption (equipping students with directly applicable skills), and straightforward development flow requiring only minimal files to get started \cite{kelhala:2024}. Applications can be easily built using C and CMake, with support extending to a wide set of architectures and microcontrollers. This stands in stark contrast with alternatives like Zephyr, which, while powerful, provides more complex tooling and documentation better suited for students with existing embedded development experience \cite{kelhala:2024}.

The development environment utilizes Visual Studio Code with the Cortex-Debug extension and CMake, providing full integration with raw Pico SDK and FreeRTOS APIs while offering simple building, flashing, and debugging capabilities through an intuitive user interface \cite{kelhala:2024}. This setup was chosen over PlatformIO, which, despite offering a complete toolchain, limited access to raw SDK functionality due to its Arduino core foundation \cite{kelhala:2024}.

The peripherals present in the TKJHAT reflect its educational design philosophy of balancing simplicity for beginners while maintaining access to low-level hardware functionality for advanced learners \cite{kelhala:2024} and are interfaced in two different ways: via General-Purpose Input/Output (GPIO) pins and Inter-Integrated Circuit ($\text{I}^2\text{C}$) buses.

Peripherals connected via $\text{I}^2\text{C}$ include:

\begin{itemize}
    \setlength\itemsep{0pt}
    \setlength\parskip{0pt}
    \item Ambient light sensor (VEML6030)
    \item Temperature \& Humidity sensor (HDC2021)
    \item Inertial measurement unit - IMU (ICM-42670, accelerator and gyroscope)
    \item Organic light emitting diode - OLED display (SSD1306)
\end{itemize}

Peripherals connected via GPIO pins include:

\begin{itemize}
    \setlength\itemsep{0pt}
    \setlength\parskip{0pt}
    \item Two light emitting diodes - LEDs: one red LED (1206 2.1V) and one red-green-blue - RGB LED (CLV1A-FKB-CK1VW1DE1BB7C3C3)
    \item A buzzer (PKM13EPYH4000-A0)
    \item A Pulse Density Modulation - PDM, Micro-Electro-Mechanical Systems - MEMS microphone (ICS-41350)
\end{itemize} 

The peripherals present that need simple, fast, one-bit control use GPIO, while peripherals that are complex sensors or multi-function chips use I²C so they can exchange structured data efficiently over a shared bus. Ideally, with the two mechanisms shown, students will also come to understand the advantages and disadvantages in both protocols and will have the knowledge to know when to choose each in real system design. This hands-on approach aligns with established teaching approaches, as physical interaction with hardware enables students to develop deeper understanding of hardware limitations and system constraints than simulation alone could provide.

The platform supports progressive learning through its Software Development Kit (SDK), which provides standardized APIs with functions to set up sensors and retrieve data \cite{kelhala:2024}. While component datasheets are heavily utilized, the SDK can be configured to either expose or hide hardware complexity, allowing instructors to adapt the platform to different expertise levels as the course progresses \cite{kelhala:2024}. Students can choose between simplified functions for basic operations or direct register access for learning low-level functionality \cite{kelhala:2024}.
